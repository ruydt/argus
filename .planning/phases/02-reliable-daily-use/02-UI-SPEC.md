## Scope Note

Phase 2 is primarily backend hardening, reliability, and testing infrastructure. The only
frontend UI change is surfacing the `normalization_status` field (MODEL-04) as a
**degraded badge on event rows** in the Events feed. Everything else in this phase is:

- Adding `normalization_status`, `normalizer_version`, `agent_version` to `frontend/src/types/events.ts`
- Writing Vitest component + hook tests for existing components (TEST-01 through TEST-03)

The design contract below is scoped accordingly. Do not introduce new layout patterns,
new pages, or new shadcn components as part of this phase.

## Existing UI Context (Do Not Change)

The existing badge patterns in `EventBadges.tsx` use:

```tsx
<Badge
  variant="outline"
  className="text-[0.68rem] text-[#888] border-white/5 bg-white/[0.04] px-[6px] py-[2px] h-auto rounded"
>
  <strong className="text-[#aaa] font-semibold mr-1">Label:</strong> value
```

The degraded badge must **not** follow this neutral pattern — it must be visually distinct
to serve as a warning indicator. See Component Contract below.

## Color

App is always dark — no light mode. Unchanged from Phase 1.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#111111` (`--bg`) | Background, sidebar surface |
| Secondary (30%) | `#191919` (`--card`) / `#1c1c1c` (`--secondary`) | Cards, secondary surfaces |
| Accent (10%) | `#a78bfa` (`--brand`) | Brand moments only — nav active indicator, create actions |
| Destructive | `#ff5f56` (`--delete`) | Destructive actions only |
| Muted foreground | `#666` (`--muted-foreground` / `--dim`) | Dimmed labels, secondary text |

### Degraded badge color — prescriptive

The degraded badge uses amber/yellow as its semantic color — distinct from the neutral
`#888` used by informational event badges, and distinct from `--delete` (`#ff5f56`) which
is reserved for destructive actions. Amber signals "something is off / needs attention"
without implying data loss (which red would imply).

| Token | Value | Role |
|-------|-------|------|
| Degraded text | `#f5a623` (`--bash`) | Reuse existing amber token — available in `src/index.css` |
| Degraded border | `rgba(245, 166, 35, 0.35)` | Semi-transparent amber border |
| Degraded bg | `rgba(245, 166, 35, 0.08)` | Low-opacity amber fill |

Rationale for reusing `--bash`: The amber `#f5a623` token already exists in the design
system for bash command events. Reusing it avoids introducing a new CSS custom property
and keeps the token count stable. The semantic contexts (bash command vs degraded payload)
do not conflict — they appear in different parts of the event row UI.

Accent reserved for: nav active indicator, create actions, mark highlights. The degraded
badge does NOT use accent (`--brand` / `#a78bfa`).

## Component Contract: Degraded Badge

### File location

`frontend/src/features/events/EventBadges.tsx` — add the degraded indicator as a new
conditional render inside the existing `EventBadges` component. Do not create a new file.

### Trigger condition

Render when `e.normalization_status === 'degraded'`. No render for `'ok'` or absent field.

### DOM output (prescriptive)

```tsx
{e.normalization_status === 'degraded' && (
  <Badge
    variant="outline"
    className="text-[0.68rem] font-semibold leading-none border-[rgba(245,166,35,0.35)] bg-[rgba(245,166,35,0.08)] text-[#f5a623] px-[6px] py-[2px] h-auto rounded"
  >
    degraded
  </Badge>
)}
```

Label is exactly the string `degraded` — lowercase, no prefix strong label, no icon.
The color communicates the semantic weight; a prefix label would be redundant.

### Placement in EventBadges

Add the degraded badge as the **first** item inside the `<div>` — before tool, source,
turn_id, and other informational badges. Degraded status is higher-hierarchy than metadata.

Updated `hasAny` guard must include `e.normalization_status === 'degraded'`:

```tsx
const hasAny =
  e.normalization_status === 'degraded' ||
  e.tool ||
  e.source ||
  // ... existing checks unchanged
```

### Interaction

Not clickable. Not expandable. Not hoverable (no tooltip). Visual indicator only.
Decision D-05 (CONTEXT.md): "Badge is a visual indicator only — not clickable/expandable."

### No global warning banner

Decision D-04 (CONTEXT.md): degraded events use a per-row badge, not a global banner.
Do not add any header-level or page-level warning for degraded events.

## Type Contract: `EventRecord` additions

Add to `frontend/src/types/events.ts` `EventRecord` interface:

```ts
normalization_status?: 'ok' | 'degraded'
normalizer_version?: string
agent_version?: string
```

These mirror the new backend `domain.NormalizedEvent` fields. Fields are optional because
existing stored events will not have them (pre-migration rows return NULL from SQLite,
which JSON-encodes as absent).

No other type changes are needed for this phase. `raw_payload` is NOT exposed in the
frontend type — it is backend-only storage, not returned in API responses in this phase.

## Playwright Smoke Test Contract

TEST-07 requires a browser smoke test that verifies the three main pages load with real
data. This is a test contract, not a UI contract — no new UI is created. Documented here
so the executor has the interaction model.

### Page assertions

| Page | Route | Assertion |
|------|-------|-----------|
| Events | `/` or `/events` | At least one event row is visible in the DOM |
| Sessions | `/sessions` | At least one session entry is visible |
| Dashboard | `/dashboard` | At least one stat value is non-zero |

### Data setup

Decision D-09 (CONTEXT.md): POST known Claude Code + Codex fixture payloads to
`/api/hook` before running assertions. No pre-seeded DB file. Fixtures are the same JSON
files used by TEST-06 normalization regression tests.

### Browser target

Chromium-only, headless. No Firefox, no WebKit. Decision D-10 (CONTEXT.md).

### CI cadence

Runs on every push/PR. Same cadence as backend/frontend unit tests. Decision D-10 (CONTEXT.md).
